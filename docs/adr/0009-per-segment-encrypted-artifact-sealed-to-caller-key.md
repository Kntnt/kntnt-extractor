# A per-segment encrypted artifact, built encrypt-as-you-go and sealed to the caller's own key

The extraction artifact is a container of independently-encrypted segments — each table dump, and each bounded part of a large file — built as the job runs, so that plaintext never lands on disk. The obvious alternative, building a plain archive and encrypting it once at the end, was considered and rejected on three grounds the project's other decisions force together. A single streaming cipher cannot resume across PHP invocations, because its running authentication state cannot be serialised between processes, yet the job is chunked across invocations by necessity (see [0007](./0007-pure-php-chunked-resumable-loopback-driven-extraction.md)); a plaintext archive built first would sit, complete, in the web-reachable uploads working directory (see [0008](./0008-working-directory-under-uploads-by-default-overridable.md)) for the whole build, protected only by an unguessable name — a bar the project accepts for audit metadata but not for a complete data dump; and `ZipArchive` cannot write through an encrypting stream in any case, since it needs a seekable file for its central directory. `ZipArchive`'s native AES was also considered and rejected: libzip's encryption support is not universal (the same portability risk that ruled out external binaries), its central directory leaves file names unencrypted, and it is weaker, older crypto than `sodium`. Encrypting each segment independently is therefore not merely a preference but what resumability, never-plaintext-at-rest and pure PHP jointly require.

The key handling is asymmetric. The caller generates an ephemeral X25519 key pair per run and sends the public key with the extraction request; for each segment the server draws a fresh random symmetric key, encrypts the segment with it, seals that key to the caller's public key with `sodium_crypto_box_seal`, stores the ciphertext and the sealed key, and zeroes the plaintext key. The server therefore never holds a key that can decrypt its own output — not in its job state, not on disk, not in memory beyond the moment of a single chunk. A symmetric key generated and kept server-side was considered and rejected because it would have to live next to the ciphertext for the whole build-and-wait window, so a read of the server's state would defeat the encryption; sealing removes any server-side key lifecycle entirely. `sodium` is bundled in PHP core, which the plugin requires at version 8.4, so this adds no portability cost.

Because the artifact is sealed to the caller, and only the caller's private key — which never leaves the caller — can open it, the artifact's confidentiality no longer depends on the secrecy or access control of the link it is fetched through. That is what lets the finished artifact be served as a plain static file (see [0004](./0004-background-job-and-one-time-download-link.md)) rather than streamed through PHP. Inside the container, table dumps are `mysqldump`-compatible SQL and files are stored by their path relative to the installation root, so a decrypted artifact restores with ordinary tools; the container's index is itself sealed, so a leaked artifact reveals nothing even about *which* tables and files were extracted.

## Consequences

- The artifact format is caller-visible and therefore bound to the API version (see [0005](./0005-github-releases-self-hosted-update-checker.md)); a change to it that callers can observe increments that version.
- The extraction request must carry a valid X25519 public key — its absence or malformation is a client error — and the caller must retain the matching private key to read the result.
- A leaked or intercepted artifact is useless without the caller's ephemeral private key, which existed only in that run's memory; the sealed index means even the list of extracted resources stays private.
- The server keeps its own separate, unencrypted job state for resume and serving bookkeeping (segment offsets, the caller's public key, the manifest of what to serve); this state holds no key that can open the artifact.
- The client does marginally more than with a server-returned symmetric key: it generates a key pair, sends the public key, and unseals each segment's key before decrypting — a few extra `sodium` calls in an already-decrypting client.
