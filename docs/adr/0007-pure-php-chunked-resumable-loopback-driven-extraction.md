# Extraction runs as pure PHP in resumable chunks, driven by a self-dispatching loopback loop with a cron watchdog

The extraction work — serialising database tables, packaging files, encrypting the result — is done in pure PHP, with no hard dependency on any external binary. Shelling out to `mysqldump`, `tar`/`gzip` or `gpg` via `exec`/`proc_open` was considered and rejected: those calls are fast and battle-tested, but managed and shared hosts frequently disable the process-spawning functions outright and often ship without the binaries at all, so a design that leaned on them would silently fail on exactly the ordinary hosting this plugin exists to serve. The plugin's whole reason to exist over "SSH plus `mysqldump`" is to run unattended through the REST API on a normal WordPress host, so portability outweighs the raw speed of shelling out. Serialisation goes through `$wpdb`, packaging and encryption through PHP's bundled `sodium` and stream functions.

Because PHP must do the heavy lifting itself, and because managed hosts kill long-running requests (an FPM `request_terminate_timeout` overrides `set_time_limit`), the job cannot assume it finishes in one PHP invocation. It runs in bounded chunks that persist across separate invocations — dump table X, append file parts N…N+k, resume where it left off — and must therefore be resumable by construction. What drives those chunks forward, unattended, is a self-dispatching loopback loop: on job creation and after each chunk, the job fires a non-blocking loopback request to itself that processes the next chunk; a scheduled WP-Cron event acts only as a watchdog that restarts a queue which has stalled, and a client's status poll opportunistically nudges the queue if nothing is currently dispatching. Its one requirement is that loopback HTTP to the site works — the same mechanism WordPress core uses to spawn its own cron — and the watchdog covers the minority of hosts where a given loopback dies.

Action Scheduler was considered and rejected as the driver: it is robust but heavy, creates its own database tables, and cuts against the lightweight, no-schema-migration posture the project takes elsewhere (see [0006](./0006-file-based-audit-log-random-name-age-rotated.md)); its runner also ultimately depends on the same cron tick. A single long-running background request was considered and rejected because the host timeouts above make "the entire uploads directory in one invocation" unreliable. A model in which the client's status polls do the work was considered and rejected because progress would be chained to the client's poll cadence — acceptable for a dedicated poller, far too slow for a large extraction driven by a generic client.

## Consequences

- The plugin needs no external binaries and works wherever WordPress runs; a `mysqldump` fast path, if ever added, would be a detected optional optimisation with a PHP fallback, never a requirement.
- Every stage of extraction must be resumable across PHP invocations, which shapes the artifact format (see [0009](./0009-per-segment-encrypted-artifact-sealed-to-caller-key.md)) and the job's on-disk state.
- The design depends on loopback HTTP as its primary driver, with a WP-Cron watchdog and a poll-nudge as fallbacks; a host with no working loopback still makes progress through the watchdog.
- This is the mechanism behind the create/poll shape [0004](./0004-background-job-and-one-time-download-link.md) presents to the client; the client sees only create, poll and download, never the loop underneath.
