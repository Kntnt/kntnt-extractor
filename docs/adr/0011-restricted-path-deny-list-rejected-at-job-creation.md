# A fixed deny-list of credential-bearing paths, enforced by rejecting the extraction request that selects one

The plugin refuses to extract files whose path matches a small, fixed family of credential-bearing patterns — `wp-config.php` and its sibling variants (backups, copies, editor droppings; `wp-config-sample.php` excepted, as it holds no secrets), `.env` and `.env.*` anywhere in the tree, and database dumps (`*.sql`, `*.sql.gz`, `*.sql.zip`) and key material (`*.pem`, `*.key`, `id_rsa*`) directly in the installation root. A `POST /extractions` whose selection contains a matching path is rejected at validation time with a client error that names every offending path; no job is created. The deny-list is one shared code constant, matched case-insensitively against install-root-relative paths, unit-tested pattern by pattern, and published in the REST contract so callers can build their exclusion sets from the same list the server enforces.

This closes a hole the plugin's own posture already implies is a hole. `GET /environment` deliberately redacts the secret-define family — `DB_PASSWORD`, the auth keys and salts are the values that never come down — yet the file layer would happily package the very files those values live in. The gap is not hypothetical: a live run observed two `wp-config.php.bak-*` files, each carrying the full secret family in clear text, packaged into the sealed container and delivered. Leaving a `.bak` beside `wp-config.php` after an edit is endemic in the wild, and the consumer's own exclusion filter is the primary fix but cannot be the only one — the server is the last party that can guarantee the site's secrets stay on the site, and an invariant delegated entirely to every client's correctness is not an invariant.

This is a security boundary, not the server-side categorisation [0003](./0003-generic-manifest-and-table-list-no-server-side-categorisation.md) rejected, and 0003 stands unchanged. What 0003 refused was selection *convenience* — classifying what a file is for (an original versus a generated thumbnail) so a client could query by category, which is one consumer's domain knowledge and a layering violation in a standalone product. The deny-list points the other way: it restricts what any caller may receive, rests on knowledge universal to every WordPress installation rather than specific to one consumer, and has the same shape as the root-containment rule 0003 itself already carries — a path that resolves outside the installation root is rejected outright, and a path that carries the site's secrets now is too. The manifest stays exactly as 0003 defines it: `GET /files` continues to list matching files unfiltered and unannotated, so the manifest remains a truthful report of what exists and baseline diffing keeps working. An optional `restricted` annotation on manifest entries was considered and rejected — that would be categorisation *in the manifest*, precisely 0003's line; publishing the pattern list in the REST contract gives callers the same information without it.

Silently skipping matching files at packaging time was considered and rejected: the caller's manifest accounting would desynchronise, and a misconfigured client would receive an artifact silently missing files instead of learning its selection is wrong. Failing fast at job creation, with the offending paths named, turns the client's misconfiguration into an explicit, actionable error. Naming the paths discloses nothing — they are paths the caller itself submitted — so the deliberately non-specific wording of the existence check's rejection is not weakened. The deny-list check runs before the existence check: whether a denied path exists is itself information the plugin has no reason to reveal, and a denied selection is wrong regardless.

## Consequences

- `POST /extractions` gains a new validation rejection, ordered before the existence check: a selection containing a restricted path fails with a client error listing every offending path, and no job is created.
- The deny-list lives in one code constant with positive and negative unit-test coverage per pattern class, and is documented in the REST contract as the normative list callers mirror when assembling exclusions.
- `GET /files` is unchanged: restricted paths are listed like any other file, with no annotation.
- A restricted path can be listed but never extracted — the one deliberate asymmetry between the manifest and the extraction endpoint, accepted as the cost of the invariant.
- The rejection is caller-visible behaviour, so shipping it increments the API version (see [0005](./0005-github-releases-self-hosted-update-checker.md)).
- Widening or narrowing the pattern family later is likewise a caller-visible contract change, not a tweak.
