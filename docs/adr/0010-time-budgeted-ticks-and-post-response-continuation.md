# Extraction ticks are time-budgeted, and the continuation nudge fires once per tick after the lock, delivery-hardened

The chunked, resumable, loopback-driven model of [0007](./0007-pure-php-chunked-resumable-loopback-driven-extraction.md) is unchanged in its essentials — extraction is still pure-PHP work packaged in bounded chunks that persist across PHP invocations, driven by a self-dispatching loopback with a WP-Cron watchdog and a poll-nudge as fallbacks. What changes is the granularity of a tick and the delivery of the continuation, because a live smoke run on a managed host exposed two ways the one-chunk-per-tick shape degrades in practice.

First, the self-loopback continuation that carries one chunk to the next never completed on that host: the tick fired its non-blocking successor, but the request never connected, so throughput collapsed to the WP-Cron watchdog's cadence — one chunk per cron cycle. On the reporting host that cadence was roughly 135 s, so an eight-chunk bootstrap job that should have finished in seconds instead crawled across eight cron cycles. The root leverage is chunks **per PHP invocation**, not the delivery of any individual loopback: trying to make the loopback itself reliable is the wrong layer. Second, the continuation nudge's `timeout => 0.01` could tear the socket down before the TCP connect even completed, and neither `blocking => false` nor a sub-second timeout reliably bounds cURL's connect/DNS phase — the synchronous resolver ignores sub-second timeouts unless `CURLOPT_NOSIGNAL` is set — so on a dead loopback the nudge could still stall the process that fired it.

## Decision

**Ticks are time-budgeted.** A tick packages as many chunks as fit in a wall-clock budget resolved through the Config seam under the knob `tick_budget` (default 15 s, deliberately well under the common 30 s FPM/PHP execution limits). The budgeted loop runs under the tick's single already-held per-job lock: the first chunk always runs — so a budget of zero is exactly one chunk per tick, the value the test suite pins so every per-chunk assertion keeps meaning what it says — and further chunks run while the job is still running and the deadline has not passed. Each chunk still saves the job, so the heartbeat stays fresh throughout the budget. The worst case — every loopback dead, no client polling — becomes one budget of packaging per watchdog cycle instead of one chunk per cycle.

**The continuation nudge fires once per tick, after the lock is released, and only when work remains.** It moves out of the per-chunk path and into `tick()`, fired after the `finally` releases the per-job lock so the successor tick can take the lock rather than no-opping against this one, and only when the job is still running — a finishing, failing, vanished, or lock-lost tick fires none. Because the nudging client disconnects almost immediately by design, the tick calls `ignore_user_abort( true )` before taking the lock and keeps packaging through that abort. The nudge's delivery is hardened through the `http_api_curl` action, which bounds the connect phase with `CURLOPT_CONNECTTIMEOUT_MS = 1000` and `CURLOPT_NOSIGNAL = 1` and uses `timeout => 1`: a healthy host finishes connect-and-send well inside that bound, and the receiving tick survives the nudger aborting at it; a dead loopback is bounded at ~1 s, which only ever costs the previous tick's process, whose own caller is long gone.

**The poll/create-side continuation runs after the response, not before it (issue #19).** Building on this budgeted `tick()` and hardened `nudge()`, `POST /extractions` and `GET /extractions/{id}` no longer couple their response latency to the continuation's network or packaging work: the continuation is registered on the `shutdown` hook, so the response is fully computed and echoed first, and a poll that should cost milliseconds is never held for the seconds a slow-to-fail loopback would otherwise add. Once the body is out, the worker prefers to drive the job in-process — it detaches from the client with `fastcgi_finish_request()` on PHP-FPM or `litespeed_finish_request()` on LiteSpeed and then runs a full budgeted `tick()`, needing no loopback at all, so a normally-polling client on a dead-loopback host advances a budget per poll rather than only at watchdog cadence. Deliberately calling `tick()` (not the staleness-gated `nudge()`) after detaching is what lets that host advance at poll cadence: driving is free for the client once detached, and the per-job lock and state guard already make a racing driver a no-op. Where the SAPI offers no detach primitive, it falls back to the same guarded, hard-bounded `nudge()`, now paid after the body is echoed — at most about a second, and never before the response. A queued or running job gets exactly one such continuation; a ready or terminal job gets none.

Alternatives considered and rejected: a loopback to `127.0.0.1` carrying a `Host:` header (fragile on SNI-routed HTTPS vhosts and edge-only hosts, and unnecessary once throughput no longer depends on the loopback at all); a loopback self-test surfaced in `GET /status` (after this change loopback health no longer gates practical throughput, so the observability is not worth a contract change now); and dropping the continuation nudge entirely (it still meaningfully accelerates the healthy-host common case).

## Consequences

- Throughput no longer depends on loopback health: a host where the self-request never connects still advances one budget of chunks per watchdog cycle, and a normally-polling client (with #19) advances a budget per poll in-process.
- The `tick()` lock / re-read / guard / `finally` structure is unchanged; only the body between the guard and the `finally` becomes the budgeted loop, and the single continuation nudge moves below the `finally`. The early returns for a terminal state, a lost lock, or a vanished job still correctly fire no nudge.
- The default `tick_budget` of 15 s stays under common execution limits; a site raises it where its host allows longer requests, or sets it to zero to restore strict one-chunk-per-tick stepping.
- The cURL hardening cannot be observed under the WordPress Playground suite, where `pre_http_request` short-circuits the transport before any cURL handle is built; it is covered by code review.
- [0007](./0007-pure-php-chunked-resumable-loopback-driven-extraction.md) is amended, not reopened: its chunked/resumable/loopback-driven model stands; only the one-chunk-per-tick granularity and the nudge's delivery hardening change here.
