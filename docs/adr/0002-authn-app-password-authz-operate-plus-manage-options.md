# Authentication via any WordPress user's Application Password; authorization gated on the Operate capability plus `manage_options`

Any WordPress user who can be granted a capability can authenticate — there is no plugin-created service account. A user calls the REST API with a WordPress Application Password, and two capabilities must both hold for any listing or extraction request to succeed: the plugin's own `kntnt_extractor_operate` capability ("may talk to this API at all") and `manage_options` (the administrator capability) for the data itself. The two compose — Operate opens the door, `manage_options` authorises the data — and neither alone is enough.

A dedicated, plugin-created user was considered and rejected: it would show up in the Users screen as an unexplained account an administrator might delete without understanding why, breaking the plugin silently. Authenticating as an ordinary WordPress user through an Application Password leaves nothing behind to be mistaken for cruft and removed.

A finer-grained authorization model was considered and rejected. That model gated each named table or file on whatever existing WordPress capability already governs it (`list_users` for the user tables, and so on), with `manage_options` as a fail-safe default for anything unmapped, so that a deliberately lower-privileged user could extract exactly the subset they already had rights to. It was dropped under YAGNI and KISS: in practice only administrators will use this plugin, and the primary consumer — the `kntnt-wp-skills` clone/pull control channel — authenticates as an administrator anyway, because pulling the options table and most other resources already required `manage_options` through that very fail-safe default. The per-resource model therefore bought a graduated, non-administrator access path that no realistic caller would travel, at the cost of a resource→capability mapping (especially fuzzy for files) and an entire capability-composition test matrix. Uniform `manage_options` is the strictest available gate and collapses authorization to a single check after input validation. Should the graduated path ever be wanted, it is cheap to reintroduce — the check sits at one seam, immediately before extraction, as a purely additive change.

The `kntnt_extractor_operate` capability is kept even though administrators already hold `manage_options`. It is the plugin's dormancy switch (see [0001](./0001-regular-plugin-dormant-by-capability-not-mu-or-timeboxed-token.md)) — the explicit, revocable "on" gate that keeps the runtime surface inert until deliberately granted, and cutting it off is how a site owner revokes API access without touching a user's administrator role. It is also the seam a future per-resource model would re-enter through. Input validation is unchanged (see [0003](./0003-generic-manifest-and-table-list-no-server-side-categorisation.md)): table names are checked against the database's actual table list and file paths are `realpath`-normalised and rejected outside the installation root, before any capability check runs — so an unknown resource is rejected regardless of the caller's capabilities.

## Consequences

- Activation grants `kntnt_extractor_operate` to the `administrator` role by default; because every request also requires `manage_options`, only administrators — or a user explicitly given both capabilities — can list or extract.
- There is no per-resource capability mapping and no fail-safe default to maintain: authorization is a single `manage_options` check performed after input validation.
- A non-administrator cannot use the API at all, even for resources a finer-grained model would have allowed. This is an accepted limitation, reversible at a single seam if it is ever needed.
- No self-healing mechanism exists for a missing Operate grant — deactivating and reactivating the plugin re-runs the activation grant, which is sufficient given there is no dedicated account left to accidentally delete.
