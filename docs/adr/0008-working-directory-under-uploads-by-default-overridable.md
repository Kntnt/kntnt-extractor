# The per-job working directory lives under the uploads directory by default, overridable to an outside-docroot path

Each extraction job needs a scratch area that is writable, persists across the job's separate PHP invocations (see [0007](./0007-pure-php-chunked-resumable-loopback-driven-extraction.md)), and holds the job's on-disk state and its growing artifact. By default this is a dedicated directory under `wp_upload_dir()`, with one randomly-named subdirectory per job, hardened with an `index.html` and an `.htaccess`/`web.config` deny as defence in depth. A strictly outside-docroot location (`sys_get_temp_dir()` or a sibling of `wp-content`) was considered and rejected as the default: it is the stronger isolation — not web-reachable at all — but WordPress guarantees no such path is writable, nor that its contents survive between requests, and on some managed and shared hosts the system temp directory is per-request, size-capped or shared. For a plugin whose portability is its reason to exist, the default must be the one location WordPress *guarantees* is writable and persistent, which is the uploads directory.

The uploads directory is ordinarily web-reachable, but that is acceptable here because the artifact is encrypted at rest and sealed to the caller's key (see [0009](./0009-per-segment-encrypted-artifact-sealed-to-caller-key.md)): a guessed or leaked hit on the random path yields only ciphertext no one but the caller can open, and the on-disk job state carries no key that could open it. The hardening files are defence in depth, not the primary control — the encryption is. For an operator who does have a reliable outside-docroot path — likely on the production sites this plugin's primary consumer targets — the location is overridable through a `KNTNT_EXTRACTOR_WORK_DIR` constant and a matching filter, following the same constant-plus-filter override pattern established in [0006](./0006-file-based-audit-log-random-name-age-rotated.md).

This settles a question [0006](./0006-file-based-audit-log-random-name-age-rotated.md) left implicit. That ADR reasoned about the audit log's location by contrasting it against "an outside-docroot working directory", written before this decision; the working directory now defaults to the uploads directory too — the same always-writable home, mitigated the same way, by a random name and by encryption rather than by server configuration. That ADR's wording has been adjusted so the contrast no longer implies the working directory is necessarily outside the docroot.

## Consequences

- A job's state and artifact live under the uploads directory by default; nothing depends on a writable path existing outside the docroot.
- The default's web-reachability is made safe by the encryption of the artifact (see [0009](./0009-per-segment-encrypted-artifact-sealed-to-caller-key.md)), not by the hardening files, which are defence in depth only.
- An operator can move the working directory outside the docroot with `KNTNT_EXTRACTOR_WORK_DIR` (or its filter) for stronger isolation, without a code change.
